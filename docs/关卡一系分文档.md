# 关卡一：一起来热身吧 · 系统分析文档

> 版本：v1.1 | 日期：2026-07-17 | 作者：liangwenbin

---

## 一、 概述

### 1.1 目标

设计并实现第一个关卡——「一起来热身吧」，包含 4 个简单动作，用户跟随完成一套约 2-3 分钟的轻度锻炼。关卡侧重**验证技术可行性**和**跑通完整闭环**（摄像头 → 动作识别 → 游戏判定 → UI 反馈）。

### 1.2 设计原则

- **动作简单、容错高**：第一个关卡不追求难度，保证 90% 的用户能顺利完成。
- **反馈即时**：每做对一个动作，UI 立刻给出视觉 + 音效反馈。
- **渐进式**：4 个动作从易到难排列，先热身再进入稍有挑战的动作。
- **无惩罚**：做不对不扣分，只记录完成次数，降低挫败感。

## 二、 关卡流程设计

### 2.1 整体流程

```
开始界面 → 3-2-1倒计时 → 动作1 → 休息 → 动作2 → 休息 → 动作3 → 休息 → 动作4 → 结算界面
```

流程说明：
- **首个动作**：3 秒倒计时后进入运动
- **后续动作**：休息结束后直接进入运动（不再倒计时）
- **运动阶段**：全部为计次制，完成目标次数后自动进入休息
- **休息阶段**（10 秒）：显示下一个动作预告
- **交互方式**：开始/重新挑战均通过双手上举保持 2 秒触发，无需点击

### 2.2 四个动作定义

| 序号 | 动作名称 | 类型 | 目标 | 难度 | 检测原理 |
|:---:|:---|:---|:---|:---|:---|
| 1 | 手臂上举 | 热身 | 完成 10 次 | ★ | 双腕高于双肩，保持 1 秒 |
| 2 | 双臂侧平举 | 上肢 | 完成 10 次 | ★★ | 双臂水平展开至肩高 + 手腕间距 > 肩宽×1.5，保持 1 秒 |
| 3 | 开合跳 | 心肺 | 完成 15 次 | ★★ | 手腕间距 > 肩宽×1.8 且 脚踝间距 > 骨盆宽×1.8 |
| 4 | 躯干扭转 | 核心 | 完成 10 次 | ★★ | 左右肩 Z 轴（深度）差值周期性正负交替 |

## 四、 核心模块详细设计

### 4.1 类型定义 (`types/index.ts`)

```typescript
// MediaPipe 返回的归一化关键点 (x, y, z 均在 0-1 范围)
interface Keypoint {
  x: number;
  y: number;
  z: number;
  visibility?: number;
}

// 33 个关键点的完整 Pose
type Pose = Keypoint[];

// 动作判定结果
interface ActionResult {
  detected: boolean;      // 本轮是否检测到动作完成
  count: number;          // 累计完成次数
  currentAngle?: number;  // 当前关节角度（用于 UI 展示）
  stage?: string;         // 状态机当前阶段（如 'up' | 'down'）
}

// 动作规则接口
interface ActionRule {
  id: string;
  name: string;
  description: string;
  icon: string;           // emoji 或图标标识
  targetType: 'reps' | 'time';  // 计次制 还是 计时制
  targetValue: number;    // 目标次数 或 目标秒数
  // 核心判定函数：输入 Pose，输出判定结果
  judge: (pose: Pose) => ActionResult;
  // 重置内部状态（切换动作时调用）
  reset: () => void;
}

// 关卡阶段
type LevelPhase =
  | 'idle'          // 未开始
  | 'countdown'     // 准备倒计时
  | 'active'        // 运动中
  | 'rest'          // 休息
  | 'complete';     // 全部完成

// 关卡状态
interface LevelState {
  phase: LevelPhase;
  currentMoveIndex: number;   // 当前第几个动作 (0-3)
  moveResults: MoveResult[];  // 每个动作的完成情况
}

interface MoveResult {
  moveId: string;
  completed: number;    // 实际完成次数
  target: number;       // 目标
  passed: boolean;      // 是否达标
}
```

### 4.3 四个动作规则 (`engine/rules.ts`)

每个规则文件导出一个工厂函数，返回 `ActionRule` 对象。内部维护闭包状态（如深蹲的 up/down 阶段计数器）。

#### 规则 1：手臂上举 (ArmRaises)

```
检测逻辑：
  - 关键点：LEFT_WRIST(15) / RIGHT_WRIST(16) 和 LEFT_SHOULDER(11) / RIGHT_SHOULDER(12)
  - 判定：双侧手腕的 Y 坐标 < 同侧肩膀的 Y 坐标 - 0.05
  - 状态机：down → up(开始计时) → completed(保持1秒) → down
  - 一次完整动作：双臂放下 → 双臂同时上举 → 保持 1 秒 → 计数 +1 → 双臂放下
  - 中途任何一只手放下：重置计时器，不计数
```

#### 规则 2：双臂侧平举 (SideArmRaises)

```
检测逻辑：
  - 关键点：LEFT_WRIST(15) / RIGHT_WRIST(16) 和 LEFT_SHOULDER(11) / RIGHT_SHOULDER(12)
  - 判定：双侧手腕 Y ≈ 同侧肩膀 Y（偏差 < 0.08），且手腕间距 > 肩宽 × 1.5
  - 状态机：down → up(开始计时) → completed(保持1秒) → down
  - 一次完整动作：双臂放下 → 双臂水平展开 → 保持 1 秒 → 计数 +1 → 双臂放下
  - 中途手臂放下或未水平：重置计时器，不计数
```

#### 规则 3：开合跳 (JumpingJacks)

```
检测逻辑：
  - 使用关键点：LEFT_WRIST(15), RIGHT_WRIST(16), LEFT_ANKLE(27), RIGHT_ANKLE(28),
                LEFT_SHOULDER(11), RIGHT_SHOULDER(12), LEFT_HIP(23), RIGHT_HIP(24)
  - 状态机：
    - 'closed': 手腕间距 < 肩宽×1.2 且 脚踝间距 < 骨盆宽×1.2
    - 'open':   手腕间距 > 肩宽×1.8 且 脚踝间距 > 骨盆宽×1.8
  - 一次完整动作：closed → open → closed
  - 注意：手脚需要同时张开/合拢，仅手或仅脚张开不算
```

#### 规则 4：躯干扭转 (TorsoTwists)

```
检测逻辑：
  - 使用关键点：LEFT_SHOULDER(11), RIGHT_SHOULDER(12)
  - 计算左右肩的 Z 轴（深度）差值：dz = leftShoulder.z - rightShoulder.z
  - dz > 0：身体左转（左肩在后、右肩在前）
  - dz < 0：身体右转（右肩在后、左肩在前）
  - 状态机：
    - 'neutral': |dz| < 阈值（面朝摄像头）
    - 'left':    dz > +阈值
    - 'right':   dz < -阈值
  - 一次完整动作：neutral → left → neutral 或 neutral → right → neutral
  - 阈值建议：0.03（归一化坐标），可根据实际测试调整
```

### 4.5 关卡状态机 (`engine/LevelFSM.ts`)

状态流转：

```
                    ┌─────────────────────────┐
                    │         idle             │
                    └────────────┬────────────┘
                                 │ 用户点击"开始"
                                 ▼
                    ┌─────────────────────────┐
                    │       countdown          │ ← 3秒倒计时
                    │    (每个动作开始前)       │
                    └────────────┬────────────┘
                                 │ 倒计时结束
                                 ▼
                    ┌─────────────────────────┐
                    │        active            │ ← 运动中，实时检测
                    │  (计次制: 达目标→结束)    │
                    │  (计时制: 倒计时→结束)    │
                    └────────────┬────────────┘
                                 │ 当前动作完成
                                 ▼
                    ┌─────────────────────────┐
                    │         rest             │ ← 10秒休息 + 下一个动作预告
                    └────────────┬────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │ 还有下一个动作    │                   │ 全部完成
              ▼                  │                   ▼
       回到 countdown            │          ┌─────────────────────────┐
                                 │          │       complete          │
                                 │          │    (结算界面)            │
                                 │          └─────────────────────────┘
```

**Transition 触发条件：**

| 当前状态 | 下一状态 | 触发条件 |
|:---|:---|:---|
| idle | countdown | 用户双手上举保持 2 秒 |
| countdown | active | 3 秒倒计时归零 |
| active | rest | 计次制：完成次数 ≥ 目标 |
| rest | active | 10 秒休息结束，还有下一个动作（直接进入，不再倒计时） |
| rest | complete | 10 秒休息结束，且已是最后一个动作 |
| 任意(运动中) | complete | 用户点击「结束」按钮 |

## 六、 数据流

### 6.1 主循环数据流

```
Camera (getUserMedia)
    │
    │ 每帧 video frame
    ▼
MediaPipe PoseLandmarker.detectForVideo()
    │
    │ Pose (33 个关键点)
    ▼
RuleEngine.judge(currentRuleId, pose)
    │
    │ ActionResult { detected, count, angle, stage }
    ▼
gameStore.updatePoseData() / completeRep()
    │
    │ 状态更新
    ▼
React 组件 re-render
    │
    ├── HUD 更新计数/进度条
    ├── Canvas 绘制骨骼
    └── 音效/特效触发 (combo 粒子等)
```

### 6.2 优化：跳帧推理

```
帧计数器 frameCount++
    │
    ├── frameCount % 2 === 0  →  跳过 MediaPipe 推理（仅绘制视频帧）
    │
    └── frameCount % 2 !== 0  →  执行完整推理 + 判定
```

每 2 帧推理一次，移动端可降至每 3 帧一次。

## 七、 边界情况与容错

| 场景 | 处理方式 |
|:---|:---|
| 用户离开摄像头范围 | HUD 显示「请回到摄像头前」，暂停计数，保留已计次数 |
| 画面中出现多人 | MediaPipe 配置 `numPoses: 1`，只取置信度最高的人 |
| 光照不足导致识别失败 | HUD 提示「光线较暗，请调整照明」，不崩溃 |
| 摄像头权限被拒绝 | 显示引导页：「请在浏览器设置中允许摄像头访问」 |
| 动作做太快导致漏检 | 状态机保证必须经过中间状态才算一次（如 squat 必须经过 down 再回到 up） |
| 移动端来电/切后台 | `visibilitychange` 事件暂停游戏，回来后从当前动作重新开始 |
| MediaPipe 模型加载失败 | 显示加载失败提示 + 重试按钮，不白屏 |

## 八、 手势交互系统

为适应无法使用鼠标键盘的场景，设计了肢体手势交互：

| 操作 | 手势 | 适用界面 |
|:---|:---|:---|
| 确认 / 开始 | 双手上举保持 2 秒 | 开始界面、结算界面 |

**实现原理**：
- 游戏循环在 idle 和 complete 阶段持续检测姿态
- 检测到双侧手腕高于肩膀时开始计时
- 保持 2 秒后触发对应操作（开始 / 重新挑战）
- UI 通过圆形进度环实时反馈保持进度
- 中途手臂放下则重置计时器

## 十、 已确认决策

1. **音效方案**：✅ 使用 Web Audio API 轻量实现（beep / countdownBeep / success 音效）
2. **动作示范**：✅ 使用文字 + emoji + 底部动作指南
3. **结算评分**：✅ 全部达标 = 3 星，3/4 = 2 星，2/4 = 1 星，<2 = 0 星
4. **交互方式**：✅ 双手上举 2 秒触发确认，去除手动按钮
5. **数据持久化**：暂不做，后续可 localStorage 存储历史成绩

| MediaPipe 在低端手机上性能不足 | 掉帧、发热 | 跳帧推理 + 降分辨率 + WASM 模式后备（不用 GPU） |
| 动作判定阈值因人体型差异不准 | 计数不准确 | 引入「校准模式」：开始前让用户站直 2 秒，自动测量身体比例来动态调整阈值 |
| iOS Safari 兼容性 | 部分功能异常 | 开发阶段就用 iOS 真机测试，避免使用不兼容 API |
| Canvas 镜像翻转逻辑混乱 | 骨骼绘制错位 | 统一在 CameraCanvas 组件内处理镜像，外部模块始终使用原始坐标 |
